About PacketGuard

Network intrusion detection, trained and evaluated openly

01What This Is

PacketGuard sorts network traffic into five categories — Normal, DoS, Probe, R2L, and U2R — using a classifier trained on NSL-KDD, the standard academic benchmark for intrusion detection. Rather than reporting a single accuracy number from a notebook, it's a working tool: upload traffic, get a classified breakdown, see what the model looked at, and run a second, independent anomaly check on the same data.

The dashboard also shows every model that was tried along the way, not just the one that shipped, so the comparison in the Compare page is the actual development history, not a highlight reel.

02Key Finding

The short version: the model gets very good at spotting the attacks it's seen plenty of, and stays weak on the rare ones — no matter how much we tuned it. A few things stood out while digging into why:

  • R2L and U2R recall never climbs past 12%, even after SMOTE oversampling and a full GridSearch pass — so this isn't a "just tune it more" problem.
  • When we looked at what the model actually pays attention to, it's leaning almost entirely on traffic-volume signals — byte counts, connection counts, how busy a service is.
  • R2L attacks don't look like that. They show up in login and session details — num_failed_logins, root_shell, num_compromised — features the model barely uses.
  • So it's not that the model can't tell R2L apart — it's that it's looking in the wrong place, because the training data gave it very few examples to learn the right place from.
  • NSL-KDD's test split makes this worse on purpose: KDDTest+ includes attack variants that never show up in training, specifically to catch models that memorize instead of generalize.
  • Which means the fix isn't a better classifier — it's better, more balanced data on what R2L and U2R traffic actually looks like.
Why this matters
This is the difference between a tuning problem and a data problem, and it changes what you'd try next. Throwing more grid search at this model won't move the needle — what would help is either collecting more R2L/U2R examples, engineering features closer to what those attacks actually change, or accepting that a supervised classifier alone won't catch them and leaning harder on the anomaly-detection path instead.
